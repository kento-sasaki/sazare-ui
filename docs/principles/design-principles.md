# Sazare UI Design Principles

> **ステータス**: 確定（2026-05-27）。変更時は [`../process/sdd-workflow.md`](../process/sdd-workflow.md) §7 のドキュメント作成サイクルに従う。

## 1. このドキュメントの目的

本ドキュメントは sazare-ui の **ミニマル・余白・装飾抑制** という美意識を、Spec レビュー時に **「この Spec は原則に従っているか」を判定できる具体的基準** へ言語化する。

sazare-ui は **Figma 等のデザインカンプを持たない**。言語化された原則とデザイントークンが唯一のビジュアル決定根拠であり（[`../README.md`](../README.md) §SDD アプローチ）、本ドキュメントは [`vision.md`](../vision.md) §4 成功基準 #1「ミニマルデザインの一貫性」を**判定可能な基準まで降ろす**責務を負う。

**Why**: カンプ不在のため、原則の言語化品質が UI 品質を直接左右する。「ミニマルにする」という感覚的指示では Spec レビューがぶれるため、各原則を「**判定の問い** ＋ **緩い目安**」の形で示し、レビュー時に OK/NG を再現的に判断できるようにする。

### 1.1 スコープ

- 視覚的なミニマル原則: ミニマル判定 / 余白 / 装飾の許容範囲 / アクセントカラー / タイポグラフィ / アイコン
- 各原則の「Spec レビューでどう判定するか」という運用基準

### 1.2 スコープ外

| 対象 | 委譲先 |
| --- | --- |
| アクセシビリティ（コントラスト比・キーボード・セマンティック HTML・ARIA） | [`accessibility.md`](./accessibility.md) |
| トークンの**具体値**（色のパレット・サイズスケール・行高・シャドウ階層の数値） | [`../tokens/*.md`](../tokens) |
| Props 命名・compound component などの API 方針 | [`api-design.md`](./api-design.md) |
| コンポーネント・ファイル・CSS 変数などの命名規則 | [`naming-conventions.md`](./naming-conventions.md) |

本ドキュメントは**「どう判断するか（原則）」を定め、「具体的な値」は tokens に委譲する**。原則がトークン設定の根拠となり、トークンが原則の数値的な裏付けとなる関係を保つ。

### 1.3 判定基準の読み方

各原則は次の構造で記述する。

- **判定の問い**: Spec レビュー時に自問するチェック項目（Yes/No で答えられる形）
- **緩い目安**: 厳密なしきい値ではなく、方向性として示す参考値（「〜程度」「最大 N を目安」）
- **OK/NG 例**: 仮想 Spec に当てたときの判定例

**Why ハイブリッド（問い＋緩い目安）**: 厳密なしきい値だけだと境界値の機械的判定に終始し美意識を失う。問いだけだと判定がぶれる。問いで「考える観点」を固定し、目安で「程度感」を共有する。目安はトークン確定後（[`../tokens/*.md`](../tokens)）に数値へ接続する。

## 2. ミニマル原則（全体を貫く判断軸）

> **designPrinciple.md §2「ミニマルなデザイン」吸収元**

すべての要素は「**なくても成立しないか**」を最初に問う。ミニマルとは「足りているのに足す」ことを避ける態度である。

**判定の問い**:

- この要素（枠線・影・背景・アイコン・区切り線など）は**機能上必要**か。装飾のための装飾になっていないか。
- 同じ情報を**より少ない要素**で伝えられないか。
- 削っても情報・操作性が損なわれないなら、削る理由がない限り削る。

**緩い目安**:

- 1 コンポーネントが持つ**視覚的装飾要素**（影・枠線・背景色・区切り線など、情報以外の装飾）は**最小限**。迷ったら削る方向に倒す。
- 「目を引かせる」要素は 1 コンポーネントにつき主役を 1 つに絞る（§5 と連動）。

**OK / NG 例**:

- ✅ OK: Button の Spec が「背景色＋ラベルのみ。枠線・影なし」を primary バリアントの既定とする。
- ❌ NG: Card の Spec が「枠線 ＋ 影 ＋ 背景の濃淡」で区切りを三重に表現する → 区切りは 1 手段で足りる（§4 と連動）。

## 3. 余白

> **designPrinciple.md §2「余白は少し広めにとり、情報の高密度化を避ける」吸収元**

余白は「埋めるべき空白」ではなく **情報をグルーピングし、ゆとりを生む設計要素** として扱う。少し広めにとり、情報の高密度化を避ける。

**判定の問い**:

- 要素間の余白は、**関連の強さ**を反映しているか（関連が強いほど近く、弱いほど離す）。
- 情報を詰め込んで密度を上げていないか。窮屈さを感じないか。
- 余白の値は [`../tokens/spacing.md`](../tokens/spacing.md) のスケールから選んでいるか（リテラル値の直書きをしていないか）。

**緩い目安**:

- 標準よりやや広めを既定とする。「狭いか広いか迷ったら広い方」。
- 余白の具体値は **spacing トークンに 100% 委譲**する。本ドキュメントは「広めに・スケールから選ぶ」という方針のみを定める。

**OK / NG 例**:

- ✅ OK: Form の Spec が「フィールド間は spacing トークンの中間段（例: `md`）、グループ間はさらに広い段（例: `lg`）」と関連の強さで余白を変える。
- ❌ NG: Modal の Spec が本文パディングに `16px` とリテラル指定する → spacing トークン経由にする（[`../process/sdd-workflow.md`](../process/sdd-workflow.md) §3.3 リテラル禁止）。

## 4. 装飾の許容範囲（影・グラデーション・枠線）

> **designPrinciple.md §2「ボタンの枠線は控えめに」「影やグラデーションなどの装飾も控えめに」吸収元**

装飾は **原則として抑制し、明確な機能的用途がある場合に限り条件付きで許可** する。「きれいに見せるため」は用途として認めない。

| 装飾 | 方針 | 許可される用途（例） | 認めない用途（例） |
| --- | --- | --- | --- |
| **影 (shadow)** | 条件付き許可 | エレベーション（重なり順）の表現。Modal・Popover・Dropdown など「浮いている」ことを示す必要がある場合 | 平面要素（Card・Button）の見栄えのための影 |
| **グラデーション** | 原則禁止 | （現時点で許可用途を定めない。必要が生じたらレビューで都度判断し本表を更新） | 背景・ボタンの装飾全般 |
| **枠線 (border)** | 条件付き許可 | 区切りが余白だけでは不十分な場合の境界明示、または input の操作可能領域の提示 | 余白で区切れる箇所への枠線の重ね掛け |

**判定の問い**:

- その装飾は上表の「許可される用途」に該当するか。該当しないなら外す。
- 同じ目的（区切り・階層表現）を**余白や配置**で達成できないか。装飾はそれが不十分なときの最終手段。
- 影の強度は [`../tokens/shadow.md`](../tokens/shadow.md) の階層から選んでいるか。

**緩い目安**:

- 影は「浮いている要素ほど強く、平面に近いほど弱く」。強度の段階と上限は shadow トークンに委譲する。
- 枠線は「主張しない」太さ・色。強調のための太枠は使わない。太さ・色の具体値は border-width / color トークン（[`../tokens/color.md`](../tokens/color.md) ほか）に委譲し、リテラル直書きはしない。

**OK / NG 例**:

- ✅ OK: Dropdown の Spec が「メニューパネルに shadow トークンの中段を付与し、浮遊を表現」。
- ❌ NG: Button primary の Spec が「立体感を出すため縦グラデーション ＋ 影」→ グラデーション禁止、平面要素への装飾影も用途外。

## 5. アクセントカラー（役割ベース）

> **designPrinciple.md §2「アクセントカラーは最小限に抑えシンプルさを追求する」吸収元**

アクセントカラーは **「1 画面に主役は 1 つ」** という役割で上限を定める。色数の固定ではなく、**役割の埋もれ防止**を基準とする。

**判定の問い**:

- 1 画面（または 1 コンポーネント）で**最も注目させたいアクション**は 1 つに絞れているか。アクセントカラーはその主役に充てているか。
- アクセントを複数の要素に使って、主役が埋もれていないか。
- セマンティックカラー（成功・警告・エラー・情報）を**アクセントと混同**していないか。両者は別軸として扱う。

**緩い目安**:

- **主役（primary アクション）= アクセントカラー 1 つ**。それ以外のアクションは中立色（グレースケール）に倒す。
- セマンティックカラーは「状態の意味伝達」のための別枠であり、アクセントの上限とは独立にカウントする。
- 色の**具体的なパレット・コントラスト比・段階**は [`../tokens/color.md`](../tokens/color.md) に委譲する。本ドキュメントは「最小限・主役を埋もれさせない」方針のみを定める。

**OK / NG 例**:

- ✅ OK: 画面の保存ボタンだけアクセントカラー、キャンセルは中立色。エラートーストはセマンティックの error 色（別軸なので可）。
- ❌ NG: 同一画面で「保存」「複製」「共有」を 3 つともアクセントカラーにする → 主役が埋もれる。1 つに絞る。

## 6. タイポグラフィ（原則のみ・値はトークンに委譲）

> **designPrinciple.md §2「シンプルなフォントファミリーを選定し、適切な行の高さ・文字間隔を設定」吸収元**

**判定の問い**:

- フォントファミリーは**シンプル**か（装飾的な書体を避けているか）。
- 行高・文字間隔は**可読性とゆとり**を確保しているか（詰めすぎていないか）。
- フォントサイズ・行高・文字間隔の値は [`../tokens/typography.md`](../tokens/typography.md) のスケールから選んでいるか。

**緩い目安**:

- フォントサイズスケール・行高・文字間隔の**具体値は typography トークンに 100% 委譲**する。本ドキュメントは「シンプル・可読性とゆとり優先」という方針のみを定める。

**OK / NG 例**:

- ✅ OK: Text コンポーネントの Spec がサイズ・行高をすべて typography トークン参照で記述。
- ❌ NG: 見出しに `line-height: 1.1` とリテラルで詰める → typography トークン経由 ＋ 可読性確保の観点で要再検討。

## 7. アイコン

> **designPrinciple.md §2「線画アイコンを中心に、スタイルとサイズを統一する」吸収元**

**判定の問い**:

- アイコンは**線画（アウトライン）スタイル**で統一されているか。塗りつぶし（フィル）スタイルを混在させていないか。
- アイコンサイズは統一された基準（後続のアイコン仕様 / spacing・typography との整合）から選んでいるか。
- アイコンは**機能を補助**しているか。装飾目的の不要なアイコンになっていないか（§2 ミニマル原則と連動）。

**緩い目安**:

- 線画中心・スタイル統一・サイズ統一を方針とする。
- アイコンセットの選定・サイズの具体値・線幅などの**詳細仕様は将来の専用ドキュメントに委譲**する（現時点では未作成。本ドキュメントは「線画中心・統一」という方針のみを定める）。

**OK / NG 例**:

- ✅ OK: Button の leading icon が線画アイコン、サイズはテキストサイズと整合。
- ❌ NG: 同一画面で線画アイコンと塗りアイコンを混在させる → スタイル統一に反する。

## 8. トークンとの接続

本ドキュメントの各原則は、後続作成のデザイントークンによって**数値的な裏付け**を得る。原則が「なぜその値か」の根拠を、トークンが「具体的にいくつか」を担う。

| 原則 | 接続するトークン | 接続のかたち |
| --- | --- | --- |
| §3 余白 | [`../tokens/spacing.md`](../tokens/spacing.md) | 「広め」をスペーシングスケールの段で具体化 |
| §4 装飾（影） | [`../tokens/shadow.md`](../tokens/shadow.md) | エレベーション階層と強度上限を定義 |
| §4 装飾（枠線） | [`../tokens/radius.md`](../tokens/radius.md), [`../tokens/color.md`](../tokens/color.md) | 枠線の角丸・色を定義 |
| §5 アクセントカラー | [`../tokens/color.md`](../tokens/color.md) | プライマリ／セマンティック／グレースケールの具体パレット |
| §6 タイポグラフィ | [`../tokens/typography.md`](../tokens/typography.md) | フォント・サイズスケール・行高・文字間隔 |
| §7 アイコン | （将来のアイコン仕様） | サイズ・線幅の具体値 |

**Why**: 原則とトークンの**一次ソースを二重化しない**ため（[`../README.md`](../README.md) §DESIGN.md の位置づけ）。原則は「判断軸」、トークンは「値」と役割を分離し、整合性管理の破綻を防ぐ。

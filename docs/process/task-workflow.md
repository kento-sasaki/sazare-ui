# Task Workflow

> **ステータス**: 確定（2026-05-17）。変更時は `sdd-workflow.md` §7 のドキュメント作成サイクルに従う。

## 1. このドキュメントの目的

本ドキュメントは sazare-ui におけるタスク管理の運用ルールを定義する。

### スコープ

- 「タスク」と Issue の対応関係
- Issue のライフサイクル（起票 → 作業中 → クローズ）
- ブランチの切り方と命名規則

### スコープ外

- コミットメッセージ規約 → グローバルの `~/.claude/rules/git-workflow.md` に準拠
- PR テンプレート・レビュー・マージ手順 → 将来追加予定
- リリース・バージョニング → `process/release.md`, `process/versioning.md`（Phase 3）

### 対象読者

- このリポジトリで作業する自分自身（将来の自分を含む）
- Claude（AI エージェント）

---

## 2. 大原則

> **1 タスク = 1 Issue = 1 ブランチ**

sazare-ui におけるあらゆる作業（ドキュメント作成・コンポーネント Spec 追加・実装・リファクタ・依存更新・設定変更など）は、必ず以下の手順を踏む。

1. GitHub Issue を起票する
2. その Issue 専用のブランチを `main` から切る
3. ブランチ上で作業し、commit する
4. PR を作成し、`main` にマージする
5. Issue をクローズする

**例外**: タイポ・誤字脱字・コメント修正など、内容に実質的影響しない極小変更は `main` へ直接コミットしてよい。判断に迷う場合は Issue 起票側に倒す。

**Why**:

- 個人開発でもタスク粒度・進捗・意図を Issue として明文化することで、将来の自分や Claude が文脈を取り戻せる
- ブランチを必ず切ることで、`main` を常に「動く・確定済み」状態に保つ
- Issue ⇄ ブランチ ⇄ PR の 1:1:1 対応により、変更追跡が機械的に可能になる

## 3. Issue のライフサイクル

### 3.1 起票

- **タイミング**: 作業着手前
- **作成方法**: `.github/ISSUE_TEMPLATE/task.yml` テンプレート、または skill `creating-github-issue` を使う
- **必須項目**: 概要 / 背景・目的 / 関連ファイル / 作業内容 / 完了条件 / 検証方法
- **Project への登録**: `sazare-ui project` に追加し、状態を `Todo` にする

### 3.2 作業中

- **タイミング**: ブランチ作成後
- **状態**: `In progress` に変更
- **作業内容**: §4 のブランチ命名規則に従ってブランチを切り、その上で作業
- **進捗の記録**: 必要に応じて Issue にコメントで進捗・意思決定を残す

### 3.3 クローズ

- **タイミング**: PR が `main` にマージされた直後
- **状態**: `Done` に変更
- **クローズ方法**: PR 本文に `Closes #<n>` を含めて自動クローズ、または `gh issue close <n>` で手動クローズ
- **完了条件のチェック**: Issue 本文の「完了条件」「検証方法」をすべて満たしていることを確認

## 4. ブランチ命名規則

### 4.1 形式

```
issue-<n>-<slug>
```

- `<n>`: Issue 番号（例: `7`）
- `<slug>`: 短い英語 kebab-case の説明（例: `sdd-workflow`）

### 4.2 例

| Issue                           | ブランチ名                    |
| ------------------------------- | ----------------------------- |
| #7 sdd-workflow.md を作成       | `issue-7-sdd-workflow`        |
| #12 Button コンポーネントを追加 | `issue-12-button-component`   |
| #25 a11y 原則ドキュメント       | `issue-25-accessibility`      |

### 4.3 ルール

- **`main` から切る**: 別ブランチからは切らない（依存タスクは Issue を分けるか、blocked-by で紐づける）
- **1 Issue = 1 ブランチ**: 同じ Issue で複数ブランチを切らない
- **マージ後の扱い**: マージ完了したブランチは削除する（GitHub の "Delete branch" を使う）
- **`<slug>` の付け方**: Issue タイトルから最重要キーワードを 2〜4 語抜く。長すぎる場合は省略

### 4.4 命名で守ること

- ASCII のみ（日本語・絵文字・スペース不可）
- kebab-case（アンダースコア・キャメルケース不可）
- すべて小文字

# SDD Workflow

> **ステータス**: 確定（2026-05-17）。変更時は §7 のドキュメント作成サイクルに従う。

## 1. このドキュメントの目的

本ドキュメントは sazare-ui を **Spec-Driven Development (SDD)** で進めるための **運用ルール** を定義する。

### スコープ

- Spec の作成プロセスとライフサイクル
- 実装と Spec の一貫性をどう保つか
- 個人開発における自己レビューの扱い
- ドキュメント作成自体のサイクル

### スコープ外

- デザイン判断の基準 → `principles/design-principles.md`
- 個別コンポーネントのテストの書き方やツール選定 → `quality/testing-strategy.md`
- Spec テンプレートの具体項目 → `components/_template.md`

### 対象読者

- このリポジトリで作業する自分自身（将来の自分を含む）
- Claude（AI エージェント）が SDD プロセスを理解する際のリファレンス

### 前提

- 個人開発プロジェクトであり、第三者レビュワーは存在しない
- Figma などのデザインカンプは存在しない（`tokens/*.md` と `principles/*.md` が唯一のビジュアル決定根拠）
- Claude を第三者役として SDD プロセスに組み込む

---

## 2. 全体フロー（俯瞰）

新規コンポーネントを 1 つ追加する場合の全体フロー。

> **前提**: 作業開始前に Issue 起票とブランチ作成を済ませること。詳細は [`task-workflow.md`](./task-workflow.md) を参照。

```
[0] Issue 起票 → ブランチ作成   ← task-workflow.md に準拠
        ↓
[1] Spec ドラフト        ← _template.md をコピー、ヒアリングして埋める
        ↓
[2] Spec レビュー         ← Claude に矛盾チェックを依頼、修正
        ↓
[3] Spec 確定 (commit)   ← components/<name>.spec.md として確定
        ↓
[4] 実装                  ← TypeScript + Panda CSS、code-reviewer agent
        ↓
[5] テスト                ← Spec ↔ テストを 1:1 翻訳
        ↓
[6] セルフレビュー         ← pre-push-reviewer agent
        ↓
[7] マージ
```

実装中に Spec の曖昧さや矛盾に気づいた場合は、実装を止めて **[2] Spec レビュー** に戻る（詳細は §4）。

## 3. 新規コンポーネント追加ライフサイクル

### 3.1 Spec ドラフト

**手順**:

1. `docs/components/_template.md` をコピーして `docs/components/<component-name>.spec.md` を作成する
2. Claude が `AskUserQuestion` で論点を提示し、ユーザーと擦り合わせる
   - 例: Anatomy をどう切るか、状態をどう列挙するか、バリアントの分類軸
3. 合意した方向性に基づいて Claude がドラフトを書く
4. 参照する原則・トークンを `principles/*.md`, `tokens/*.md` から引用して埋め込む

**ドラフトに含める項目**:

`components/_template.md` の確定項目（README.md §「Spec テンプレートの確定項目」参照）。

**いきなり完成版を書かない**。ヒアリング → ドラフト → レビュー → 確定のサイクルを必ず通す（詳細は §7）。

### 3.2 Spec レビュー・確定

**Claude に第三者役として以下のチェックを依頼する**:

- Props 命名が `principles/naming-conventions.md` に従っているか
- アクセシビリティ要件が `principles/accessibility.md` を満たしているか
- 参照しているトークンが `tokens/*.md` に存在するか
- 「状態」表と「インタラクション」表に翻訳漏れがないか（§5 の 1:1 翻訳の前提）
- 不変条件・例外ケースの記述に抜けがないか
- 他のコンポーネント Spec と矛盾していないか

**確定の条件 (DoD)**:

- `_template.md` の全項目が埋まっている
- 上記レビューで指摘された矛盾・抜けがすべて解消されている
- ユーザーが内容に合意している

確定後、Spec を `docs/components/<component-name>.spec.md` として commit する。

### 3.3 実装

**前提**: Spec が確定し commit されていること。

**手順**:

1. Spec の Props API・Anatomy・状態・バリアントを TypeScript の型レベルに落とす
2. Panda CSS の recipe を書く（**スタイル値は必ず `tokens/*.md` 経由**、リテラル禁止）
3. Spec の不変条件をコード上の assertion または型制約で表現できないか検討する
4. 実装中に Spec の曖昧さ・矛盾に気づいた場合は実装を中断し、§4 の Spec 変更フローに従う
5. 実装完了後、`code-reviewer` agent を必ず通す

### 3.4 テスト（Spec ↔ テスト 1:1 翻訳）

**原則**: Spec の「状態」と「インタラクション」を手動で 1:1 にテストケースへ翻訳することで、Spec 違反を機械的に検出可能にする。

- Spec の「状態」表: **1 状態 = 1 テストケース**
- Spec の「インタラクション」表: **1 行 = 1 テストケース**
- Spec の「不変条件」: **1 つ = 1 assertion**
- Spec の「アクセシビリティ」: a11y テストで担保

翻訳漏れは Spec レビュー時（§3.2）にもチェックされるが、実装時にも再度確認する。

**具体的なテストの書き方・ツール選定・カバレッジ基準は `quality/testing-strategy.md` を参照**。本ドキュメントは「Spec とテストが 1:1 で対応する」という運用ルールのみを定める。

## 4. Spec 変更時のフロー

Spec は確定後も変更されうる。以下の 2 ケースを区別する。

### 4.1 実装中に Spec の曖昧さ・矛盾に気づいた場合

1. **実装を一旦止める**（中途半端な状態で先に進めない）
2. Spec を更新する
3. 影響範囲を Claude にチェックさせる（他コンポーネントの Spec、原則、トークンへの波及）
4. Spec 確定後に実装を再開する

### 4.2 既存コンポーネントの Spec を意図的に変更する場合

1. 変更の動機を Spec の末尾（または別ログ）に記録する
2. §3.2 と同じ Spec レビュー手順を通す
3. 既存テストへの影響を洗い出す（1:1 翻訳のどの行が変わるか）
4. 破壊的変更を含む場合は、`process/versioning.md`（Phase 3 で着手予定）のポリシーに従う

## 5. Spec と実装の一貫性保証

Spec と実装の一貫性は **3 層** で担保する。

| 層     | 仕組み                                                            | タイミング             |
| ------ | ----------------------------------------------------------------- | ---------------------- |
| 一次   | Spec ↔ テストの 1:1 翻訳（§3.4）                                  | 実装と並行             |
| 二次   | Spec のテストシナリオ節と実テストファイルの 1:1 対応              | 実装完了時             |
| 三次   | `pre-push-reviewer` agent によるセルフレビュー                    | マージ前               |

**自動マッピングツールは作らない**。Spec の状態・インタラクションを手動で 1 行ずつテストへ翻訳することで、Spec の曖昧さが翻訳時に必ず顕在化する。これが SDD の核となる強制力である。

**Why ツール化しない**:

- カンプ不在の前提下では、Spec の言語化品質が UI 品質を直接左右する。ツール化すると Spec の曖昧さが温存される。
- 個人開発の規模では、手動翻訳のコストは支払い可能な範囲。YAGNI。

## 6. 自己レビューの扱い（個人開発特有）

個人開発であるため、第三者レビュワーは存在しない。代替として **Claude を第三者役** として活用する。

### 6.1 自己レビューのタイミング

| タイミング           | やること                                | 主な手段                       |
| -------------------- | --------------------------------------- | ------------------------------ |
| Spec ドラフト直後    | 齟齬・抜け・矛盾の指摘                  | Claude に Spec を読ませて質問  |
| 実装完了時           | コード品質・セキュリティ・型安全性      | `code-reviewer` agent          |
| マージ前             | 変更全体のセルフレビュー                | `pre-push-reviewer` agent      |

### 6.2 Claude に第三者役を任せる際の前提

- Claude には **「自分の書いたものを批判してほしい」** と明示する（賛成意見ではなく反対意見を引き出す）
- 指摘は鵜呑みにせず、原則・トークン・既存 Spec と照合して採否を判断する
- 「Claude が問題なしと言ったから安全」とは見なさない

### 6.3 自己レビューで防ぎたい主な事故

- 命名規則違反（`principles/naming-conventions.md` との乖離）
- アクセシビリティ要件の見落とし
- 既存トークンを使わずにリテラル値を書く
- 状態・インタラクションの翻訳漏れ
- 既存コンポーネントとの API 不整合

## 7. ドキュメント作成サイクル（メタ）

本リポジトリのすべてのドキュメント（本ドキュメント自身を含む）は、以下のサイクルで作成する。

```
[1] ヒアリング   ← Claude が論点を AskUserQuestion で提示
       ↓
[2] ドラフト     ← 合意した方向性に基づいて Claude がドラフト
       ↓
[3] レビュー     ← ユーザーと一緒にレビュー、不足・違和感を洗い出す
       ↓
[4] 確定         ← 修正反映後、commit
```

### 7.1 守るべき原則

- **いきなり完成版を書かない**。曖昧な点は必ずヒアリングで潰す。
- ヒアリング履歴は `docs/README.md` の「ヒアリング履歴」セクション、または各ドキュメント末尾に残す。
- ドラフトには「⚠️ ステータス」を明記し、レビュー前後を区別できるようにする。

### 7.2 Claude の役割

- ヒアリング: `AskUserQuestion` で 3〜4 個の論点を選択肢付きで提示
- ドラフト: 合意内容を文章化、参照リンクを埋め込む
- レビュー: ユーザーから「直して」と言われた箇所のみ修正、勝手に範囲を広げない

# ADR 0001: ヘッドレスUI + スタイリングの技術スタック

Button / Dialog のプロトタイプ検証を経て、Ark UI + Panda CSS を採用する。

## コンテキスト

- sazare-ui は **React 専用**の UI ライブラリ／デザインシステム（マルチフレームワーク対応は評価不要）。
- スタイリングは「**ランタイムで CSS を生成しない**」ことを制約とする。Emotion / styled-components のようなランタイム CSS-in-JS、およびそれらを内部で使う Chakra UI / MUI (Material UI) は除外。一方、Tailwind CSS や CSS Modules のように「CSS-in-JS ではないがゼロランタイム」な方式も候補に含む（ビルド時に静的 CSS を生成する方式であれば手段は問わない）。
- ソロ開発のため、**参照実装から移植・応用できること**を重視する（GitHub 上の実装例や OSS コンポーネントライブラリを調査し、車輪の再発明を避ける）。
- デザイントークン・variant を型安全に扱えることが望ましい（sazare-ui はデザイントークンの実装が v0.1 の必須スコープであるため）。
- 主要コンシューマーである mabiki の既存スタック（Radix UI + Tailwind CSS v4 + CVA）は、**意図的に評価軸に含めない**（sazare-ui の技術選定はゼロベースで行う方針を別途確認済み）。
- 2026-05-29 に一度ゼロベース評価を行い（Issue #25 / PR #26、`docs/architecture/tech-stack.md`）「Ark UI + Panda CSS」を第一推奨としたが、当該ドキュメントは 2026-07-10 の docs 再構成で削除され、**その結論は引き継がない**。今回はその後の状況変化（後述の Base UI 等）も踏まえ完全に再調査する。

## 決定ドライバー（評価軸）

1. React 専用であることに対する適合度（マルチ FW 対応は評価上の加点にならない）
2. アクセシビリティ・キーボード操作・ARIA の実装品質
3. ゼロランタイム（ビルド時に静的 CSS を生成する）であること
4. 型安全なデザイントークン／variant 管理のしやすさ
5. 参照実装（移植・応用できる実装例）の有無
6. ライブラリの最新性・メンテナンス状況（2026-07-10 時点で npm レジストリを実測）

## 検討した候補：L1（ヘッドレスUI、挙動・アクセシビリティ）

| 候補 | 最新版 (2026-07-10 実測) | 概要 | 強み | 弱み |
|---|---|---|---|---|

| **Ark UI**（+ Zag.js） | 5.37.2（2026-06-08 公開） | Zag.js（状態機械）ベースのマルチFWヘッドレス。`data-scope`/`data-part`/`data-state` でスタイリングフック。 | 状態機械による堅牢な挙動。Panda CSS との組み合わせに Park UI という参照実装がある。 | マルチFW対応の強みは React 専用の sazare-ui では活きない。概念量・codegen がやや重い。 |

| 候補 | 最新版 (2026-07-10 実測) | カテゴリ | 概要 | 強み | 弱み |
|---|---|---|---|---|---|
| **Panda CSS** | 1.11.4（2026-06-27 公開） | ゼロランタイム CSS-in-JS | 静的解析でビルド時に atomic CSS を生成。型安全なトークン/recipe。 | recipe・variant・トークンの型安全性が高く、コンポーネントライブラリのProps API実装と相性が良い。Park UI という参照実装がある。 | `panda codegen` のビルドステップが必須。完全に動的な値は苦手。 |
| **StyleX** | 0.19.0（2026-06-16 公開） | ゼロランタイム CSS-in-JS | Meta製。Babelプラグインでatomic CSSへコンパイル。決定的なスタイルマージ。 | atomic CSS生成による最小出力、型安全。 | コンポーネントライブラリ向けの軽量な参照実装が乏しい（Astryxは自前L1込みの重量級）。 |

| **CSS Modules** | （ビルドツール機能。パッケージ本体のバージョン概念なし） | スコープ付きCSS（CSS-in-JSではない） | ビルド時にクラス名をローカルスコープ化。素のCSSをそのまま書く。 | 依存最小・10年以上安定した実績。学習コストが最も低い。 | 型安全なトークン/recipeの仕組みがなく、variant管理は別途（例: CVA）自前で組む必要がある。 |
| **vanilla-extract** | 1.21.1（2026-06-30 公開） | ゼロランタイム CSS-in-JS | TypeScriptでスタイルを記述し、ビルド時にプレーンCSSへコンパイル。 | 配布物がプレーンCSSになるため消費者非結合が最も強い。 | コンポーネントライブラリ向けの軽量な参照実装が乏しく、手配線が多くなりがち。 |

## 検討したスタック（L1 + L2 のペア）

L1とL2は技術的には `data-*` 属性経由で疎結合に組み合わせ可能だが、参照実装の有無で実装コストが大きく変わるため、ペア単位でも評価する。

| スタック | 参照実装 | コメント |
|---|---|---|
| Ark UI + Panda CSS | Park UI (Ark+Panda) | 移植性が最も高い。Ark のマルチFWの強みは今回活きない点は要考慮。 |
| Base UI + Panda CSS | なし（新しい組み合わせ） | 最新のL1と型安全なL2の組み合わせだが、参照実装なしで手配線が増える。 |
| Base UI + Tailwind CSS v4 | shadcn-ui（2026年7月以降のデフォルト） | 最大のコミュニティ・参照コード量。ただし所有型前提の設計を npm 型ライブラリに転用する際の規律が必要。 |
| Radix UI + CSS Modules | なし（自前構成） | 依存最小・最も「枯れた」構成。型安全なトークン/recipeは別途自前で用意。 |

## プロトタイプ検証（2026-07-11実施）

Button と Dialog を実装し、3つのスタック候補を実測した（`prototypes/` 配下、使い捨てコード）。「コンシューマー側での導入のシンプルさ」を評価軸に追加した。

| 観点 | Ark UI + Panda CSS | Ark UI + CSS Modules | Base UI + CSS Modules |
|---|---|---|---|
| 実装行数（Button+Dialogのみ、config/test除く） | **49行**（Button 17 + Dialog 32） | 166行（Button 61 + Dialog 105） | 147行（Button 52 + Dialog 95） |
| 別途必要な初期投資 | panda.config.ts 111行（recipe基盤。以降の全コンポーネントで再利用可能な一次投資） | なし | なし |
| A11y（フォーカストラップ／Escape／フォーカス復帰／ラベル自動配線） | 全てArk UI既定・自前配線ゼロ | 全てArk UI既定・自前配線ゼロ | 全てBase UI既定・自前配線ゼロ。ただし `aria-modal` は使わず、`aria-hidden` + inert方式で外側要素を無効化してモダリティを実現する設計（Ark/Radix系のaria-modal方式と異なるが、いずれもARIA準拠） |
| ゼロランタイム | 確認済み（`dist/index.css` 17KB、preflight込み。JSにランタイム注入コード無し） | 確認済み（`dist/index.css` 1.31kB） | 確認済み（`dist/index.css` 1.13KB） |
| 型安全なvariant/トークン | ◎ コンパイル時に検証、補完あり | ✗ 文字列インデックス、typoは実行時まで検出不可 | ✗ 同左 |
| 消費者(mabiki)側の追加作業 | `npm i` + `import "@sazare-ui/components/style.css"` 1行 + import。追加バンドラ設定なし | 同左 | 同左（`package.json`への追加も1行のみ） |
| ライブラリ側の固有事項 | `panda codegen` がビルド前提（ライブラリ内部の工程、消費者は無関係）。Ark UI/Panda/clsx等は全てdistにバンドルされ、externalはreact/react-domのみ | 同上（clsxのみバンドル） | `Dialog.Viewport` という新しいアナトミーパーツが必須。合成は`asChild`ではなく`render` prop（`useRender`）で、カスタム子コンポーネントに forwardRef + props展開が必要 |

**重要な発見**: 「コンシューマー側での導入のシンプルさ」は3候補間で**差が出なかった**。react/react-dom以外の全依存をライブラリ側にバンドルし静的CSSを1ファイルに出力する構成にすれば、L1/L2の選択に関わらず消費者側は `npm install` + CSS import 1行 + コンポーネントimportで済む。この軸は当初の想定に反して決定を左右しなかった。

一方、**実装行数の差は決定的**だった。CSS Modules勢は1コンポーネントあたり147〜166行を要するのに対し、Panda CSSは一次投資（recipe基盤111行）さえ済ませれば以降は数十行で済む。v0.1で35〜40コンポーネントを実装する計画（`docs/requirement.md` および会話ログの決定事項）を踏まえると、この差はコンポーネント数に比例して拡大する。加えて型安全なトークン/variantはtypoを実行時ではなくコンパイル時に検出でき、デザイントークンが必須スコープであるsazare-uiの性質と噛み合う。

**未検証の組み合わせ**: Base UI + Panda CSS は今回テストしていない。Base UI は新しく（2026年6月にv1.6.0）、Panda との組み合わせ参照実装（Park UIのようなもの）が無いため、仮に採用するなら追加の手配線が必要になる可能性が高い。

## 決定

**Ark UI + Panda CSS を採用する。**

理由:
1. v0.1スコープ（コンポーネント約35〜40個）の規模で、Panda CSSのrecipe基盤による一次投資が実装行数を大幅に削減する（CSS Modules比で1/3程度）。
2. 型安全なトークン/variantが、デザイントークンを必須スコープとするsazare-uiの性質と合致する。
3. A11y品質はArk UI/Base UIどちらも実測で満点（自前配線ゼロ）のため差がつかない。
4. 「コンシューマー側の導入シンプルさ」は評価軸として機能しなかった（3候補とも同等）。
5. Park UIという参照実装が存在し、ソロ開発での参照・移植コストが最も低い。

Base UI + Panda CSSという組み合わせは未検証のまま残ったが、ユーザー確認の上でこの決定を確定した。

この決定は前回（2026-05-29、削除済み）の結論と結果的に一致するが、今回は独立した再調査とハンズオンの実測に基づいて到達した点が異なる。

## 影響・懸念（未解決事項）

- sazare-ui の配布モデルは [0002](./0002-package-distribution-model.md) で決定済み（npm private公開）。
- Base UI は安定版としての実績期間が短いため、長期的なメンテナンス継続性は unconfirmed。
- **実装時の注意（プロトタイプ検証で判明）**:
  - `typescript@latest` は現在v7系（Go実装ベータ）を指し、`vite-plugin-dts`が非対応。3候補すべてで踏んだため、ライブラリ側のビルド基盤に共通する問題。実装時は `typescript` のバージョンを明示的に固定すること。
  - Base UI の npm パッケージ名は `@base-ui/react`。似た名前の `@base-ui-components/react` は旧称で `1.0.0-rc.0` のまま更新が止まっている別パッケージなので注意。
  - Base UI の Dialog は `aria-modal` 属性を使わず、外側要素に `aria-hidden` + inert 相当の無効化を適用する方式でモダリティを実現する（Ark UI/Radix系の `aria-modal` 方式と異なる）。テストや将来の他コンポーネントの実装でこの前提の違いを踏まえること。

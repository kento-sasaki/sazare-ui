# ADR 0004: パッケージ構成

## コンテキスト

sazare-ui は「デザイントークン」と「Reactコンポーネント」の両方を v0.1 のスコープに含む（要件15番）。これらを単一パッケージにまとめるか、複数パッケージに分割するかを決める必要がある。パッケージマネージャーは pnpm（要件13番）と既に決まっている。

## 決定ドライバー（評価軸）

1. 将来的に他フレームワークや他アプリでトークンのみを再利用する可能性
2. ソロ開発における実装・保守コスト（過剰な抽象化を避ける : YAGNI）
3. [0003](./0003-release-and-versioning.md) の Changesets によるバージョン協調のしやすさ

## 検討した選択肢

| 選択肢 | 強み | 弱み |
|---|---|---|
| 単一パッケージ（`@sazare-ui/components` 一つにトークンもコンポーネントも含める） | 構成がシンプル。想定コンシューマーが mabiki のみの現時点では分割の必然性がない | 将来分割が必要になった場合に移行コストが発生する |
| pnpm workspace で `packages/tokens` と `packages/components` に最初から分割 | トークンをフレームワーク非依存の資産として独立させておける。将来 Vue 等の他フレームワークや他アプリでトークンのみ再利用する場合に備えられる | 現時点でコンシューマーが mabiki のみのため、分割の恩恵はまだ顕在化していない |

## 決定

**pnpm workspace で `packages/tokens` と `packages/components` に最初から分割する。**

筆者（Claude）の推奨は「単一パッケージ」であったが、ユーザーの明示的な判断により分割構成を採用した。

## 影響・懸念（未解決事項）

- `packages/tokens` の実体（CSS変数出力のみか、[0001](./0001-headless-ui-and-styling-stack.md) で決まる L2 スタイリング手段のトークン定義形式に依存するか）は技術スタック確定後に決める。
- 将来的にパッケージが増える場合（例: アイコンパッケージ）の命名規則は本ADRでは決めない。

# ADR 0007: テスト戦略

## コンテキスト

要件には Chromatic（ビジュアル回帰）と ESLint/Prettier/cSpell の記載はあるが、ユニットテスト・アクセシビリティ自動検査・E2Eについては記述がなかった。旧 `implementationPlan.md`（削除済み）では Vitest + Playwright + アクセシビリティ自動化を掲げていたが、これを v0.1 からそのまま引き継ぐかは未確認だった。

## 決定ドライバー(評価軸)

1. ヘッドレスUIライブラリ（[0001](./0001-headless-ui-and-styling-stack.md) で選定予定）採用により、複雑な挙動ロジック自体はライブラリ側でテスト済みであることが多い
2. ソロ開発における初期実装コストとのバランス
3. v0.1は「広く浅く」実装し `draft` タグで段階的に調整する方針（要件16番）との整合性

## 検討した選択肢

| 選択肢 | 強み | 弱み |
|---|---|---|
| Vitest（単体）+ Storybook a11y addon（axe-core） | コンポーネントのロジック・propsに応じた出力を最低限カバーしつつ、A11yはStorybookカタログ上で自動検査できる。E2Eを持たない分、初期実装コストを抑えられる | 実ブラウザでの結合的な検証（E2E）は持たない |
| Vitest + axe-core + Playwright（E2E含め全部） | テストピラミッドを最初から完備でき品質は最も高い | ソロ開発での初期実装コストが増える |

## 決定

**Vitest（単体テスト）+ Storybook a11y addon（axe-core ベース）** を採用する。E2E（Playwright）は v0.1 では見送る。

## 影響・懸念（未解決事項）

- 将来 E2E が必要になるタイミング（例: 複合コンポーネントが増えた時点）は別途判断する。
- カバレッジ目標値は本ADRでは定めない。

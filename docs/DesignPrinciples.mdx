# デザイン原則

## この文書の位置づけ

多くの公開デザインシステムの「デザイン原則」ページは、次のいずれかを目的に書かれている。

1. **UIの体験品質を形容詞で規定する**：Carbon / Salesforce Lightning Design System / Adobe Spectrum / Apple HIG / Ant Design などがこれに当たる。「Clarity」「Beauty」「Focused」のような言葉で、複数のデザイナーが同じ見た目や手触りに収束するよう揃える
2. **サービスを作る側の振る舞いを命令形で規定する**：GOV.UK / U.S. Web Design System などがこれに当たる。多数の省庁やチームが同じプロセスで意思決定するよう揃える

どちらも共通しているのは、複数の人間の意思決定を一致させることが目的である点だ。

sazare-ui にはこの前提が当てはまらない。作り手は一人であり、揃えるべき「他者の判断」が存在しない。加えて sazare-ui は[個別コンシューマー（mabiki）のブランドや見た目の温度感を意図的に持ち込まない汎用の基盤](./adr/0013-decorative-color-token-sourcing.md)として設計されており、「見た目の形容詞」を原則の中心に据える根拠となるブランドそのものを持たない。

そこでここでの原則は、他者との合意形成のためではなく、時間をおいて文脈を失った自分自身が一人で高速かつ一貫した判断を下すための基準として書く。数ある構造の中では、価値観と「Xより先にYを満たす」という優先順位をペアで示す Atlassian Design System の書き方が、単発の意思決定を導くという点で最も参考になった。以下ではまず sazare-ui が拠って立つ**価値観**をいくつか定義し、次にその価値観を実務判断に落とし込むための具体的な**原則**を定義する。

迷ったときは、まずこの4つの原則のどれに当てはまるかを考える。当てはまらない判断が出てきたら、それは原則が足りていないというサインなので、後から追記してよい。個別の技術スタックやトークン仕様といった具体的な決定は、この文書の後半（「現在の技術的な結論」）および各 [ADR](./adr/) を参照する。

---

## 価値観

sazare-ui の判断は、次の4つの価値観に支えられている。

- **強いデフォルト**：選択肢の多さではなく、一つの強いデフォルトを信頼する
- **汎用な土台**：sazare-ui は特定プロダクトの所有物ではなく、複数プロダクト共通の土台である
- **半年後の自分**：今の自分の記憶ではなく、半年後に文脈を失った自分が読むことを前提にする
- **判断の再利用**：全プロダクトで同じ見た目を強制するのではなく、同じ状況には同じ考え方で応じる

以下の「原則」は、この4つの価値観をそれぞれ実務の判断基準に落とし込んだものである。

## 原則

<Markdown>
{`
| 価値観 | 原則（優先順位） |
| --- | --- |
| 強いデフォルト | 選択肢を増やすことより、いまあるデフォルトを強くすることを優先する |
| 汎用な土台 | 個別プロダクトの要望を満たすことより、複数プロダクトに共通する基盤であり続けることを優先する |
| 半年後の自分 | 実装の速さより、半年後に文脈を失った自分が読めることを優先する |
| 判断の再利用 | その場の判断より、過去に出した結論を記録して再利用できることを優先する |
`}
</Markdown>

### 1. 強いデフォルト（Strong Defaults）

**原則：** 選択肢を増やすことより、いまあるデフォルトを強くすることを優先する。

新しい props やバリアントを増やしたくなったら、先に「今のデフォルトを直せないか」を疑う。レビュアーのいない個人開発では、増やした選択肢を将来の自分がすべて覚えていられる保証がない。選択肢は資産ではなく、半年後の自分に対する負債候補として扱う。

- **NG：** 要望が来るたびに `size="sm" | "md" | "lg" | "compact" | "tiny"` のようにバリアントを積み増す
- **OK：** 要望の本質を聞き直し、本当に必要な2〜3種類に絞る。残りは他のprops（spacingなど）の組み合わせで表現できないか検討する
- **実例：** デザイントークンに対応するpropsは任意のCSS値ではなくトークンキーの列挙型のみを許容する（[ADR 0011](./adr/0011-token-constrained-props.md)）。`className` / `style` を公開propsから排除し、レイアウトの合成は `Box` / `Stack` に一本化する（[ADR 0012](./adr/0012-no-classname-style-props.md)）。どちらも「選べる幅を広げる」のではなく「用意された強いデフォルトの組み合わせで表現させる」という同じ判断の適用である

### 2. 汎用な土台（Generic Foundation）

**原則：** 個別プロダクトの要望を満たすことより、複数プロダクトに共通する基盤であり続けることを優先する。

あるプロダクトだけのための見た目や挙動は、sazare-ui本体ではなくプロダクト側の拡張レイヤーに置く。2つ以上のプロダクトで同じ要求が出て初めて、本体への昇格を検討する。「今それを使うのは mabiki だけ」は、sazare-ui に取り込む理由にはならない。

- **NG：** mabiki固有の概念（例:「紙色に馴染むテキスト色」）や専用コンポーネントをそのまま sazare-ui に持ち込む
- **OK：** 値（色相など）は既存プロダクトから借りてよいが、命名やAPIは sazare-ui 独自の汎用的な語彙に翻訳し直す
- **実例：** 装飾色トークンの色相はmabikiの藍パレットを流用しつつ、トークン名は `colors.action.solid` のような汎用命名にし、mabiki固有の概念は持ち込まなかった（[ADR 0013](./adr/0013-decorative-color-token-sourcing.md)）。「値は借りてよいが、語彙は借りない」という線引きが、この原則を実務に落とした具体的な判断である

### 3. 半年後の自分（Future-Self Clarity）

**原則：** 実装の速さより、半年後に文脈を失った自分が読めることを優先する。

命名・型定義・READMEやコメントは、他人に説明するつもりで書く。「今は分かるから後で書く」を許さない。一人開発は暗黙知が積もりやすく、後から見て「なぜこうしたか」が分からなくなることが一番のリスクである。

- **NG：** 意味の異なる2つの概念を、見た目が同じというだけで同じprop名に押し込む（例:「大きさ」を表す `size` に見出しレベルの意味も持たせる）
- **OK：** 迷った時点で手を止め、型で意味を区別できる名前に直してからコミットする
- **実例：** 意味的な構造（見出しレベル `as`）と見た目のサイズ（`size`）をpropsとして分離し、型レベルで区別する（[ADR 0010](./adr/0010-typography-component-api-pattern.md)）。半年後の自分が `size="lg"` を見出しレベルの指定だと誤読しないようにするための設計である

### 4. 判断の再利用（Reusable Judgment）

**原則：** その場の判断より、過去に出した結論を記録して再利用できることを優先する。

たとえば「エラー表示はどう見せるか」という問いに、プロダクトごと・その時々の気分で毎回ゼロから考え直さない。過去に出した答えを再利用し、変える場合はその理由を記録する。記録がなければ、半年後の自分は結論だけを見て「なぜそう決めたか」を再現できない。

- **NG：** 同じ種類の判断（例: props をトークン列挙にするか任意値も許すか）を、コンポーネントごとにその場の感覚で決め直す
- **OK：** 一度出した判断はADRとして記録し、次に同種の判断が必要になったら過去のADRを引用して踏襲する（変える場合は新しいADRとして理由を残す）
- **実例：** 意思決定はADRとして `docs/adr/` に記録し、Storybookをコンポーネントドキュメントの唯一のSSoTとする（[ADR 0008](./adr/0008-docs-structure.md)）。この文書自体も、原則という「繰り返し使う判断基準」を一度書いて使い回すための実践であり、この原則の実例そのものである

## 現在の技術的な結論

上記の原則を土台に、現時点で sazare-ui が採用している技術的な結論を簡潔に記す。比較検討の詳細は各ADRを参照。

- **トークン体系：** Primitive → Semantic の2層モデル。propsはトークンキーの列挙型のみを許容し、任意のCSS値は不可（[ADR 0011](./adr/0011-token-constrained-props.md)）
- **コンポーネント構成：** ヘッドレスUI（Ark UI）+ ゼロランタイムスタイリング（Panda CSS）で構成する（[ADR 0001](./adr/0001-headless-ui-and-styling-stack.md)）
- **アクセシビリティ：** WCAG 2.2 Level AA を目標とし、Storybook a11y addon（axe-core）でコンポーネント単位に自動検査する。各コンポーネントのmdxに「アクセシビリティ」節を設ける（例: [Heading.mdx](../packages/components/src/typography/Heading/Heading.mdx)）（[ADR 0007](./adr/0007-testing-strategy.md)）
- **ドキュメント運用：** Storybookをコンポーネントカタログ・ドキュメントの単一のSSoTとし、カテゴリ別サブディレクトリで整理する（[ADR 0008](./adr/0008-docs-structure.md), [ADR 0009](./adr/0009-storybook-sidebar-organization.md)）

## スコープ外

一般的なデザインシステムに見られる要素のうち、以下は現在のゴール（mabikiでの実利用開始）に照らして採用しない。将来ゴールが変わった際に見直す前提。

- **多言語対応（i18n）/ RTL**：mabikiは日本語単一言語での実利用を前提とするため、現時点ではスコープ外とする
- **テーマ切替（ダーク/ライトモード等）**：mabikiの現在の実利用要件に含まれないため見送る
- **マルチフレームワーク対応**：sazare-uiはReact専用として設計しており、Vue/Svelte等への対応は行わない（[ADR 0001](./adr/0001-headless-ui-and-styling-stack.md)）
- **W3C DTCG形式への完全準拠**：Style Dictionary等の外部変換パイプラインを必要とする運用は、npm経由でReactコンポーネントとして配布する現在の配布モデル（[ADR 0002](./adr/0002-package-distribution-model.md)）には過剰であるため見送る。トークンはTypeScriptオブジェクトとして直接定義・配布する
- **E2Eテスト（Playwright）**：v0.1では見送る（[ADR 0007](./adr/0007-testing-strategy.md)）
- **UIの見た目・体験を規定する形容詞的な原則（Clarity / Beauty 等）**：[ADR 0013](./adr/0013-decorative-color-token-sourcing.md)により、sazare-uiは特定コンシューマーのブランド・見た目の温度感を意図的に持ち込まない方針であるため、参照すべきブランドを持たない現時点ではスコープ外とする。将来 sazare-ui 自身が視覚的な作風を持つ判断をした場合に見直す

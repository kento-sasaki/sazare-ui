# Sazare UI Vision

> ⚠️ **ステータス**: ドラフト（2026-05-27）。レビュー後に確定する。変更時は [`process/sdd-workflow.md`](./process/sdd-workflow.md) §7 のドキュメント作成サイクルに従う。

## 1. このドキュメントの目的

本ドキュメントは sazare-ui の **目的・成功基準・非ゴール** を明文化し、後続のあらゆる判断（デザイン原則・デザイントークン・コンポーネント Spec・公開準備）の **判断基準（North Star）** とする。

迷ったときは「これは sazare-ui の目的に沿うか」「成功基準に貢献するか」「非ゴールに踏み込んでいないか」を本ドキュメントだけで判断できる粒度を目指す。

**Why**: sazare-ui は「個人開発専用だが本格レベル」というやや矛盾しやすい位置づけで運営される。方向性を確定しないと、原則・トークン・コンポーネント Spec の優先順位判断がぶれる。

---

## 2. 一次目的とスコープ

sazare-ui は **個人開発専用** の React UI ライブラリだが、**「本格」レベル** で開発する。「個人用だから雑でいい」ではなく、デザインシステムとしての一貫性・品質を OSS 同等レベルで担保する。

### 2.1 一次目的

> **自分自身の個人開発プロジェクトで再利用できる、一貫性のある UI ライブラリを持つこと。**

- 開発効率の向上（毎回 UI を作り直さない）
- デザインの一貫性の確保（プロジェクト間でルックの統一）
- 将来の個人開発プロジェクトでの再利用性

### 2.2 npm 公開の位置づけ

npm 公開は **副次的な目的** として視野に入れる（§6 参照）。一次目的を犠牲にしてまで公開を急がない。

### 2.3 スコープ

- **デザインアセット**: 色・タイポグラフィ・スペーシングなどのデザイントークン
- **UI コンポーネント**: ボタン・フォーム要素・モーダル・フィードバック系など、個人開発で頻出する基本コンポーネント
- **ガイドライン**: コンポーネントの使用方法（Storybook を活用）

## 3. 想定ユーザー（読者）

sazare-ui のドキュメント・コードの読者は次の 2 者を想定する。

| 読者 | 位置づけ |
| --- | --- |
| **自分自身**（将来の自分を含む） | 一次の利用者・意思決定者。ライブラリを使い、Spec を書き、実装する |
| **Claude（AI エージェント）** | SDD プロセスの第三者役・実装補助。ドキュメントを文脈として読み、Spec レビューや実装を担う |

**Why**: 第三者レビュワーが存在しない個人開発のため、Claude を第三者役として SDD プロセスに組み込む（[`process/sdd-workflow.md`](./process/sdd-workflow.md) §6）。ドキュメントは「将来の自分」と「Claude」の両方が文脈を取り戻せる粒度で書く。

npm 経由の第三者利用者は **現時点では一次読者として想定しない**（公開時に改めて検討する）。

## 4. 成功基準

sazare-ui が「成功している」と言える状態を、方向性レベルで定義する。**測定可能な細目（コントラスト比の具体値・カバレッジ閾値・DoD など）は後続ドキュメントに委譲する。**

| # | 成功基準 | 概要 | 細目の委譲先 |
| --- | --- | --- | --- |
| 1 | **ミニマルデザインの一貫性** | 余白重視・装飾控えめという美意識が全コンポーネントで貫かれている | [`principles/design-principles.md`](./principles/design-principles.md) |
| 2 | **WCAG 準拠のアクセシビリティ** | コントラスト比・キーボード操作・セマンティック HTML が基準を満たす | [`principles/accessibility.md`](./principles/accessibility.md) |
| 3 | **トークン / API の一貫性** | デザイントークンが活用され、Props 命名・API 設計が統一されている | [`tokens/*.md`](./tokens), [`principles/api-design.md`](./principles/api-design.md) |
| 4 | **技術基準の達成** | TypeScript strict・全コンポーネントの Storybook 文書化・Spec とテストの 1:1 対応 | [`quality/review-checklist.md`](./quality/review-checklist.md), [`quality/testing-strategy.md`](./quality/testing-strategy.md) |

**Why この 4 つ**: `docs/implementationPlan.md` §5「品質基準」の「ミニマル / アクセシビリティ / 一貫性」と技術基準を、プロジェクト全体の North Star として昇格させたもの。個別コンポーネントの達成度はこの 4 軸で測る。

## 5. 非ゴール（やらないこと）

スコープの肥大化を防ぐため、**明示的にやらないこと** を定める。下記に該当する要望は、原則として却下するか別プロジェクトに切り出す。

| 非ゴール | 理由 |
| --- | --- |
| **複雑なデータ可視化（グラフ・チャート）** | 専用ライブラリの領域。個人開発で頻出する基本 UI に集中する |
| **Figma 等デザインカンプの作成・維持** | カンプは持たない。**言語化されたデザイントークン・原則が唯一のビジュアル決定根拠**（[`README.md`](./README.md) §SDD アプローチ） |
| **汎用 OSS としてのユースケース網羅** | 「自分の個人開発で使うもの」を本格品質で作る。あらゆるユースケースを網羅する汎用ライブラリ化は目指さない |

**判断テスト**: 機能要望が来たら「個人開発で自分が頻繁に使うか」「ミニマルの美意識を保てるか」を問う。例: 「複雑なグラフコンポーネントが欲しい」→ 非ゴール（データ可視化）に該当するため却下。

## 6. npm 公開ポジショニング

| 項目 | 方針 |
| --- | --- |
| **優先度** | 副次。一次目的（個人再利用）を優先し、公開を急がない |
| **温度感** | 「視野に入れる」。コンポーネント実装が進み、API が安定してから公開準備に着手する |
| **想定読者** | 公開時に第三者利用者を読者として追加検討する（§3） |
| **準備内容** | パッケージ化・SemVer・リリースフローは Phase 3 で着手（[`README.md`](./README.md) §着手順序） |

**Why 副次**: 公開を一次目的にすると、汎用性・網羅性（§5 非ゴール）に引きずられて個人開発での使い勝手が犠牲になりやすい。「自分が本格品質で使えるものが、結果として公開にも耐える」という順序を守る。
